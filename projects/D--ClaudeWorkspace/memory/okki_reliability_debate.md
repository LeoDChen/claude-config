# OKKI 项目可靠性问题：两个 AI 对话的交叉分析

## 背景

用户连续在两个独立对话中启动 OKKI 项目，结果都不满意：
- 对话 A（本对话）：同步后发现会话数从 490 掉到 466
- 对话 B（另一个新对话）：输出结果"还是有问题"，被 gen_s3_rules.js 规则引擎污染

用户将两个对话的分析互相喂给对方，要求交叉验证。

## 两个 AI 各自发现的问题

### 对话 A（本对话）发现
1. **会话数波动（490→466，-24 人）**：OKKI 虚拟列表窗口变化，老旧会话被清出
2. **全量覆盖写入风险**：`sync_okki.js` 每次覆盖 `tm_conversations_latest.json`，长期累积丢失数据
3. **提出架构修复**：改为累计合并（tm_conversations_master.json），只增不删
4. **gen_s3_rules.js 残留**：磁盘上存在规则引擎脚本（后用 `rm` 删除）

### 对话 B 发现
1. **gen_s3_rules.js 规则引擎污染**：250+行正则匹配把 465 人全判成"需求探索"和"搁置"两个阶段
2. **CLAUDE.md 旧版本包含该脚本**：AI 看到流水线里有就跑了
3. **修复**：删除 gen_s3_rules.js，CLAUDE.md 加硬约束禁止脚本生成 s3
4. **提出三层防护**：启动自检清单 + s3 分布异常检测 + 脚本数量门禁
5. **低估了累积丢失风险**：认为"覆盖写入不是根因，OKKI 没返回所以合并不进来"

## 交叉验证结论

| | 对话 A | 对话 B | 实际 |
|---|---|---|---|
| gen_s3 规则引擎污染 | 之前没强调 | 抓住了 | 两者判对，B 更紧急 |
| 累积丢失风险 | 抓住了 | 低估了 | A 对——多次同步叠加丢失 |
| 覆盖写入是根因 | 是长期根因 | 认为不是 | A 对 |
| 订单匹配/分类链路 | 看了 | 没看 | A 更完整 |
| 防守机制 | 未设计 | 设计了三层 | B 补充得好 |

## 两个问题都要修

1. **gen_s3 修复（紧急）**：影响当前看板质量 · 已完成（删脚本 + CLAUDE.md 硬约束）
2. **累积合并（基础）**：影响长期数据可靠性 · 待实施（改 sync_okki.js 保存逻辑）
3. **三层防护（防守）**：防止新对话 AI 再次绕路 · B 已写入 CLAUDE.md

## 关键教训

- 不同 AI 对话对同一问题的诊断可能互补，需要人类交叉验证
- 脚本残留 + CLAUDE.md 旧指令 = 新对话 AI 必然踩坑
- 防守不能只靠文档，必须物理删除 + 硬约束 + 异常检测三层
