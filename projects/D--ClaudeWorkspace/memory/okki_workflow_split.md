# /okki 完整工作流

## 触发时用户提供

1. `/okki` 命令
2. 全量订单原文（从阿里订单列表页直接复制粘贴）
3. r1/r9 脚本自动抓取 OKKI 数据（脚本负责，AI 不干预）

## 执行流程

### 第1步：订单入库
1. 解析用户粘贴的订单原文 → 更新 `data/orders_snapshot.json`
2. **管道自动同步**：`pipeline_okki.js` 在 init 前自动将 snapshot 转换为 `data/orders_latest.json`（旧格式，供 init 读取）
3. `init_factor_baselines.js` 的 `correctStageByOrder` **全阶段覆盖**（活跃订单优先于已关闭订单）自动纠正阶段
4. 不再依赖 `sync_orders.js` 抓取订单数据——订单数据源 = 用户提供

### 第2步：多 Agent 深度分析（同前）
- 每批 5-8 个 Agent，拿到一批再下一批
- Agent 分析时读取聊天原文回传摘要
- **Agent 产出驱动因子/下一步/产品需求，不产出阶段标签**

### 第3步：管道输出 HTML
1. `gen_dashboard_528.js` 生成看板（阶段已由订单数据纠正，Agent 驱动因子已合并）
2. 自动打开浏览器

## 订单状态映射参考

| 用户提供(snapshot) | 管道转换(latest) | 客户阶段(colKey) |
|------|------|------|
| 待买家付款 | 待收款 | 待付款 |
| 待卖家发货 | 待发货 | 待发货 |
| 待买家确认收货 | 已发货 | 在途 |
| 订单完成 | 交易成功 | 已签收 |
| 订单关闭 | 交易取消 | 未成交 |
| 待卖家接单 | 未确认 | 待付款（极小概率，不单独列） |
| 待买家支付预付款 | 待收款 | 待付款 |
| 待买家支付尾款 | 待收款 | 待付款（防护映射） |

活跃订单（待收款/待发货/已发货/未确认）优先于关闭订单（交易成功/交易取消）。
优先级：已发货:4 > 待发货:3 > 待收款:2 > 未确认:1

## 输出质量门

- [ ] 管道日志显示 "已同步订单数据: XX 条"
- [ ] `stage_corrections.json` 修正数 > 0
- [ ] 交叉验证：有订单客户阶段错误数 = 0
- [ ] **驱动因子/下一步/产品规格**：全部客户均由 Agent 分析产出，订单快照不替代

## 关键原则

- **订单快照只给阶段标签**（待付款/待发货/在途/已签收），其余全部由 Agent 产出
- **阶段纠正 ≠ Agent 白分析**：Agent 仍要深读聊天记录提取驱动因子、产品需求、下一步动作
- 产品规格以聊天记录为准，订单上的产品链接可能是壳
- OKKI trails 的订单状态可能过期，不信任
- 最终 HTML 看板不嵌入聊天原文，保持体积可控
